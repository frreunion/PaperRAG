## 3. 当前达到的效果

当前系统已经能跑通论文 RAG 助手的最小闭环：

```text
上传 PDF
  -> 保存文件
  -> 解析 PDF 文本
  -> 提取基础元信息
  -> 切分 chunk
  -> 生成 fake embedding
  -> 建立检索索引
  -> 创建聊天会话
  -> 用户提问
  -> 检索相关 chunk
  -> 生成回答
  -> 返回 citations
```

也就是说，当前项目已经不是纯文档或静态 demo，而是一个具备前后端、接口、数据模型、检索、问答和引用展示的可运行 MVP。

适合用于：

- 作品集初版展示。
- Agent/RAG 岗位面试讲解。
- 后续接入真实 LLM、真实 embedding 和 pgvector 的工程基础。
- 继续扩展成科研论文管理工具。

## 5. 当前主要缺陷

### 5.1 AI 能力仍是 fake 实现

当前 embedding 和 LLM 都是离线 fake 实现：

- `FakeEmbeddingClient` 使用确定性哈希向量。
- `FakeLLMClient` 只是根据第一个检索片段生成简单回答。

影响：

- 可以验证工程链路。
- 不能代表真实问答质量。
- 中文语义检索能力有限。
- 不能真正完成复杂论文理解。

后续需要接入：

- OpenAI / Claude / Gemini / 本地兼容 OpenAI API 的 LLM。
- OpenAI embedding / BGE / Jina / text-embedding 等真实 embedding 模型。
- 可选 rerank 模型。

### 5.2 数据库默认使用 SQLite，不是真正 pgvector

当前后端默认使用 SQLite，并将 embedding 存为 JSON。

影响：

- 本地开发简单。
- 适合 MVP 和测试。
- 不适合大规模论文库。
- 无法使用 PostgreSQL + pgvector 的高效向量索引能力。

后续需要：

- 添加 Alembic。
- 切换 PostgreSQL。
- 启用 pgvector extension。
- 将 embedding 字段迁移为 vector 类型。
- 建立 ivfflat 或 hnsw 索引。

### 5.3 PDF 解析能力基础

当前使用 `pypdf` 解析文本型 PDF。

缺陷：

- 不支持扫描版 PDF OCR。
- 对双栏论文、公式、图表、脚注顺序处理有限。
- 表格和图片内容无法理解。
- 章节识别依赖简单正则。

后续可升级：

- 引入 GROBID 做论文结构解析。
- 引入 Marker、Nougat 或 OCR 工具处理扫描版。
- 单独提取 figure/table caption。
- 改进双栏阅读顺序。

### 5.4 RAG 策略较简单

当前检索策略是：

```text
fake vector score * 0.75 + keyword score * 0.25
```

缺陷：

- 没有真正 BM25。
- 没有 query rewrite。
- 没有 rerank。
- 没有 conversational query reformulation。
- 没有上下文压缩。
- 没有检索失败诊断。

后续需要：

- 接入 BM25 或 PostgreSQL full-text search。
- 加 rerank。
- 加 query rewrite。
- 对检索结果做去重和压缩。
- 建立 Recall@K 评测集。

### 5.5 Citation 机制可用但仍需增强

当前优点：

- citation 由后端从检索 chunk 映射生成。
- 模型不能自由生成 citation 元数据。
- quote 来自真实 chunk 文本。

不足：

- 当前引用粒度是 chunk，不是精确句子。
- 前端还不能跳转到 PDF 原页位置。
- 没有 PDF 高亮。
- 没有校验回答中每个关键事实是否都有引用。

后续需要：

- 保存句子级 offset。
- 前端接入 PDF viewer。
- 点击 citation 跳转到页码。
- 对回答做 citation coverage 检查。

### 5.6 前端还处于 MVP 工作台水平

当前前端能操作主流程，但仍有不足：

- 没有真实 PDF 预览器。
- 没有聊天历史列表。
- 没有批量上传。
- 没有标签、收藏、项目文件夹。
- 没有 Markdown 编辑/导出。
- 没有更细的处理进度。
- 移动端可用，但体验未做深度优化。

### 5.7 Docker Compose 还不是完整生产级部署

当前 `docker-compose.yml` 已定义 PostgreSQL、Redis、backend、frontend。

缺陷：

- 后端默认仍使用 SQLite。
- Redis 当前未真正用于后台队列。
- 没有 Celery/RQ worker。
- 没有数据库 migration 初始化流程。
- 没有生产环境反向代理配置。

后续需要：

- 后端切换为 PostgreSQL DSN。
- 添加 worker 服务。
- 添加 Alembic migration。
- 添加初始化 pgvector 的 SQL。
- 添加 Nginx 或 Caddy。

### 5.8 测试覆盖还偏少

当前测试覆盖了核心服务的最小行为：

- health API。
- chunking。
- retrieval。
- RAG citation。
- 前端基础渲染。

不足：

- 上传 PDF API 测试还不完整。
- 总结和对比接口测试还未覆盖。
- 没有真实端到端浏览器测试。
- 没有真实 PDF fixture。
- 没有数据库 repository 层完整测试。

## 6. 当前适合展示的亮点

在作品集或面试中，可以重点讲：

1. **端到端 RAG 工程闭环**  
   从 PDF 上传到检索问答和 citation 展示，链路完整。

2. **Citation 后端绑定设计**  
   不让模型自由编造来源，而是通过 `[S1]` 编号映射真实 chunk。

3. **模型供应商可替换**  
   通过 `EmbeddingClient` 和 `LLMClient` 抽象，后续可以替换真实模型。

4. **离线可测试**  
   fake embedding 和 fake LLM 让测试不依赖真实 API Key。

5. **前后端完整交付**  
   有 FastAPI、React、测试、Docker 配置、文档和评测脚本。

6. **清晰的后续演进路径**  
   已经识别出 pgvector、OCR、rerank、PDF viewer、评测集等下一阶段方向。

## 8. 阶段结论

当前项目已经完成“论文 RAG 助手 MVP”的工程骨架和主流程闭环，具备可运行、可测试、可演示的基础。

但它还不是生产级论文助手。当前版本更准确的定位是：

> 一个完整的 RAG 工程原型，已经打通上传、解析、切块、检索、问答、引用和前端展示流程；下一阶段重点是接入真实模型、升级向量数据库、增强 PDF 解析和扩大测试覆盖。
